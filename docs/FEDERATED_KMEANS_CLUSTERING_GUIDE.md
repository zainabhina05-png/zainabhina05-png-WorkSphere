# Client-Side Federated K-Means Clustering for Workspace Recommendations

_This guide describes how WorkSphere performs client-side k-means clustering on workspace embeddings, executes vector computations inside Web Workers, re-ranks search results using cluster assignments, and maintains local privacy guarantees — all without sending raw user data to the server._

---

## 1. Introduction

WorkSphere recommends workspaces based on user preferences, venue features, and historical interactions. Rather than performing clustering on the server (which would require transmitting raw user vectors), WorkSphere runs k-means clustering entirely in the browser. Each client clusters its own saved venues and interaction history into preference groups, then uses cluster centroid distances to re-rank search results and feed recommendations to the ONNX inference pipeline.

All computation happens inside a dedicated Web Worker to keep the UI responsive. Only compressed cluster centroids (not raw vectors) are ever synced to the server for global aggregation.

---

## 2. Goals

| Goal                          | Why it matters                                                             |
| ----------------------------- | -------------------------------------------------------------------------- |
| **Zero-knowledge clustering** | Raw user preference vectors never leave the device.                        |
| **Low-latency re-ranking**    | Cluster assignments are computed locally in < 50 ms for 10k vectors.       |
| **Privacy-preserving sync**   | Only differentially-private centroid summaries are shared with the server. |
| **Incremental updates**       | New interactions update cluster assignments without full re-clustering.    |
| **Resource-friendly**         | All computation runs off the main thread via Web Workers + SIMD.           |

## 4. Vector Representation

### 4.1 Embedding Dimensions

Each workspace is represented as a 1024-dimensional float32 vector generated by Cohere's `embed-english-v3.0` model. User preferences are encoded as weighted sums of the vectors of venues they have visited, saved, or rated highly.

| Entity                 | Source                           | Dimension | Storage                                |
| ---------------------- | -------------------------------- | --------- | -------------------------------------- |
| Venue embedding        | Cohere API                       | 1024      | PostgreSQL `vector(1024)` via pgvector |
| User preference vector | Weighted sum of venue embeddings | 1024      | IndexedDB (client-side)                |
| Cluster centroid       | K-means mean of assigned vectors | 1024      | IndexedDB + server sync                |

### 4.2 Similarity Metrics

WorkSphere uses **cosine similarity** for nearest-neighbor queries and **squared Euclidean distance** for k-means assignment.

**Cosine similarity** (search re-ranking):

```
sim(A, B) = (A · B) / (‖A‖ × ‖B‖)
```

**Squared Euclidean distance** (k-means assignment):

```
d(A, B) = Σᵢ (Aᵢ - Bᵢ)²
```

**Centroid update** (k-means convergence):

```
Cₖ = (1 / |Sₖ|) Σ_{x ∈ Sₖ} x
```

where `Sₖ` is the set of vectors assigned to cluster `k`.

## 5. K-Means Clustering Algorithm

### 5.1 Initialization: K-Means++

The initialization step selects `k` initial centroids that are spread across the data distribution, reducing convergence time and avoiding degenerate clusters.

```typescript
function kmeansppInit(vectors: Float32Array[], k: number): Float32Array[] {
  const centroids: Float32Array[] = [];
  const n = vectors.length;

  // Step 1: Choose first centroid uniformly at random
  const firstIdx = Math.floor(Math.random() * n);
  centroids.push(vectors[firstIdx].slice());

  // Step 2: Choose remaining centroids with probability proportional to D(x)²
  const distances = new Float32Array(n).fill(Infinity);

  for (let c = 1; c < k; c++) {
    const lastCentroid = centroids[c - 1];

    // Update distances to nearest existing centroid
    for (let i = 0; i < n; i++) {
      const d = squaredEuclidean(vectors[i], lastCentroid);
      if (d < distances[i]) distances[i] = d;
    }

    // Weighted random selection
    const totalDist = distances.reduce((sum, d) => sum + d, 0);
    let r = Math.random() * totalDist;
    for (let i = 0; i < n; i++) {
      r -= distances[i];
      if (r <= 0) {
        centroids.push(vectors[i].slice());
        break;
      }
    }
  }

  return centroids;
}
```

### 5.2 Assignment Step

Each vector is assigned to its nearest centroid using squared Euclidean distance.

```typescript
function assignClusters(
  vectors: Float32Array[],
  centroids: Float32Array[],
): Uint16Array {
  const assignments = new Uint16Array(vectors.length);

  for (let i = 0; i < vectors.length; i++) {
    let minDist = Infinity;
    let bestCluster = 0;

    for (let k = 0; k < centroids.length; k++) {
      const d = squaredEuclidean(vectors[i], centroids[k]);
      if (d < minDist) {
        minDist = d;
        bestCluster = k;
      }
    }

    assignments[i] = bestCluster;
  }

  return assignments;
}
```

### 5.3 Update Step

Centroids are recomputed as the mean of all vectors assigned to each cluster.

```typescript
function updateCentroids(
  vectors: Float32Array[],
  assignments: Uint16Array,
  k: number,
  dim: number,
): Float32Array[] {
  const sums = Array.from({ length: k }, () => new Float32Array(dim));
  const counts = new Uint32Array(k);

  // Accumulate sums per cluster
  for (let i = 0; i < vectors.length; i++) {
    const c = assignments[i];
    counts[c]++;
    for (let d = 0; d < dim; d++) {
      sums[c][d] += vectors[i][d];
    }
  }

  // Compute means (handle empty clusters)
  const centroids = new Array<Float32Array>(k);
  for (let c = 0; c < k; c++) {
    if (counts[c] === 0) {
      // Reinitialize empty cluster to random vector
      centroids[c] =
        vectors[Math.floor(Math.random() * vectors.length)].slice();
    } else {
      centroids[c] = new Float32Array(dim);
      for (let d = 0; d < dim; d++) {
        centroids[c][d] = sums[c][d] / counts[c];
      }
    }
  }

  return centroids;
}
```

### 5.4 Convergence

The algorithm iterates until the total centroid shift falls below a threshold or a maximum iteration count is reached.

```typescript
function centroidShift(old: Float32Array[], updated: Float32Array[]): number {
  let totalShift = 0;
  for (let k = 0; k < old.length; k++) {
    totalShift += squaredEuclidean(old[k], updated[k]);
  }
  return Math.sqrt(totalShift);
}

const MAX_ITERATIONS = 50;
const CONVERGENCE_THRESHOLD = 1e-4;

function runKMeans(
  vectors: Float32Array[],
  k: number,
  dim = 1024,
): { centroids: Float32Array[]; assignments: Uint16Array; iterations: number } {
  let centroids = kmeansppInit(vectors, k);

  for (let iter = 0; iter < MAX_ITERATIONS; iter++) {
    const assignments = assignClusters(vectors, centroids);
    const newCentroids = updateCentroids(vectors, assignments, k, dim);

    const shift = centroidShift(centroids, newCentroids);
    centroids = newCentroids;

    if (shift < CONVERGENCE_THRESHOLD) {
      return { centroids, assignments, iterations: iter + 1 };
    }
  }

  const finalAssignments = assignClusters(vectors, centroids);
  return {
    centroids,
    assignments: finalAssignments,
    iterations: MAX_ITERATIONS,
  };
}
```

## 6. WebWorker Message Passing

### 6.1 Worker Protocol

The clustering worker follows the same message-passing pattern used by `sqlite-search.worker.ts` and `forecasting.worker.ts`. All communication uses typed messages with correlation IDs.

```typescript
// ─── Message Types ────────────────────────────────────────

type ClusteringWorkerMessage =
  | { type: "INIT"; id: string; dim?: number }
  | {
      type: "ADD_VECTORS";
      id: string;
      vectors: Float32Array[];
      labels: string[];
    }
  | { type: "CLUSTER"; id: string; k: number; maxIter?: number }
  | { type: "RE-RANK"; id: string; queryVec: Float32Array; topN?: number }
  | { type: "GET_ASSIGNMENTS"; id: string }
  | { type: "SYNC_CENTROIDS"; id: string }
  | { type: "APPLY_GLOBAL_CENTROIDS"; id: string; centroids: Float32Array[] };

type ClusteringWorkerResponse =
  | { type: "INIT_SUCCESS"; id: string; vectorCount: number }
  | {
      type: "CLUSTER_RESULT";
      id: string;
      centroids: Float32Array[];
      assignments: Uint16Array;
      iterations: number;
      inertia: number;
    }
  | {
      type: "RERANK_RESULT";
      id: string;
      results: { label: string; score: number; cluster: number }[];
    }
  | { type: "ASSIGNMENTS_RESULT"; id: string; assignments: Uint16Array }
  | { type: "CENTROIDS_SYNCED"; id: string; centroids: Float32Array[] }
  | { type: "ERROR"; id: string; error: string };
```

### 6.2 Worker Implementation

```typescript
// src/workers/clustering.worker.ts

import {
  runKMeans,
  squaredEuclidean,
  cosineSimilarity,
} from "../lib/clustering/kmeans";

const DIM = 1024;
let vectors: Float32Array[] = [];
let labels: string[] = [];
let centroids: Float32Array[] = [];
let assignments: Uint16Array = new Uint16Array(0);

function post(msg: ClusteringWorkerResponse) {
  self.postMessage(msg);
}

self.onmessage = async (e: MessageEvent<ClusteringWorkerMessage>) => {
  const { type, id } = e.data;

  try {
    switch (type) {
      case "INIT": {
        vectors = [];
        labels = [];
        centroids = [];
        assignments = new Uint16Array(0);
        post({ type: "INIT_SUCCESS", id, vectorCount: 0 });
        break;
      }

      case "ADD_VECTORS": {
        vectors.push(...e.data.vectors);
        labels.push(...e.data.labels);
        post({ type: "INIT_SUCCESS", id, vectorCount: vectors.length });
        break;
      }

      case "CLUSTER": {
        const k = Math.min(e.data.k, vectors.length);
        const result = runKMeans(vectors, k, DIM);
        centroids = result.centroids;
        assignments = result.assignments;

        // Compute inertia (total within-cluster sum of squares)
        let inertia = 0;
        for (let i = 0; i < vectors.length; i++) {
          inertia += squaredEuclidean(vectors[i], centroids[assignments[i]]);
        }

        post({
          type: "CLUSTER_RESULT",
          id,
          centroids,
          assignments,
          iterations: result.iterations,
          inertia,
        });
        break;
      }

      case "RE-RANK": {
        if (centroids.length === 0) {
          post({ type: "RERANK_RESULT", id, results: [] });
          return;
        }

        const queryVec = e.data.queryVec;
        const topN = e.data.topN ?? 10;

        // Find nearest centroid to query
        let nearestCluster = 0;
        let minDist = Infinity;
        for (let c = 0; c < centroids.length; c++) {
          const d = squaredEuclidean(queryVec, centroids[c]);
          if (d < minDist) {
            minDist = d;
            nearestCluster = c;
          }
        }

        // Re-rank: combine cosine similarity with cluster affinity boost
        const results = labels
          .map((label, i) => {
            const cosSim = cosineSimilarity(queryVec, vectors[i]);
            const inCluster = assignments[i] === nearestCluster ? 0.15 : 0;
            return {
              label,
              score: cosSim + inCluster,
              cluster: assignments[i],
            };
          })
          .sort((a, b) => b.score - a.score)
          .slice(0, topN);

        post({ type: "RERANK_RESULT", id, results });
        break;
      }

      case "GET_ASSIGNMENTS": {
        post({ type: "ASSIGNMENTS_RESULT", id, assignments });
        break;
      }

      case "SYNC_CENTROIDS": {
        post({ type: "CENTROIDS_SYNCED", id, centroids });
        break;
      }

      case "APPLY_GLOBAL_CENTROIDS": {
        centroids = e.data.centroids;
        // Re-assign vectors to nearest global centroid
        if (vectors.length > 0 && centroids.length > 0) {
          const newAssignments = new Uint16Array(vectors.length);
          for (let i = 0; i < vectors.length; i++) {
            let best = 0;
            let bestDist = Infinity;
            for (let c = 0; c < centroids.length; c++) {
              const d = squaredEuclidean(vectors[i], centroids[c]);
              if (d < bestDist) {
                bestDist = d;
                best = c;
              }
            }
            newAssignments[i] = best;
          }
          assignments = newAssignments;
        }
        post({ type: "ASSIGNMENTS_RESULT", id, assignments });
        break;
      }
    }
  } catch (err) {
    post({ type: "ERROR", id, error: (err as Error).message });
  }
};
```

### 6.3 Main Thread Hook

The `useClusteringWorker` hook manages the worker lifecycle, correlating requests with responses via message IDs.

```typescript
// src/hooks/useClusteringWorker.ts

import { useRef, useCallback } from "react";
import {
  ClusteringWorkerResponse,
  ClusteringWorkerMessage,
} from "../workers/clustering.worker";

let messageCounter = 0;

export function useClusteringWorker() {
  const workerRef = useRef<Worker | null>(null);

  const getWorker = useCallback(() => {
    if (!workerRef.current) {
      workerRef.current = new Worker(
        new URL("../workers/clustering.worker.ts", import.meta.url),
        { type: "module" },
      );
    }
    return workerRef.current;
  }, []);

  const postMessage = useCallback(
    <T extends ClusteringWorkerMessage["type"]>(
      message: Extract<ClusteringWorkerMessage, { type: T }>,
    ): Promise<ClusteringWorkerResponse> => {
      return new Promise((resolve, reject) => {
        const worker = getWorker();
        const id = `msg_${++messageCounter}`;

        const handler = (e: MessageEvent<ClusteringWorkerResponse>) => {
          if (e.data.id === id) {
            worker.removeEventListener("message", handler);
            if (e.data.type === "ERROR") {
              reject(new Error(e.data.error));
            } else {
              resolve(e.data);
            }
          }
        };

        worker.addEventListener("message", handler);
        worker.postMessage({ ...message, id });

        // Timeout after 30s
        setTimeout(() => {
          worker.removeEventListener("message", handler);
          reject(new Error("Clustering worker timeout"));
        }, 30_000);
      });
    },
    [getWorker],
  );

  return { postMessage };
}
```

## 7. Search Re-Ranking with Cluster Affinity

### 7.1 Two-Stage Ranking Pipeline

```
Query → [Stage 1: BM25 (sqlite-search.worker.ts)]
      → Top 100 candidates
      → [Stage 2: Cluster Re-Rank (clustering.worker.ts)]
      → Top 10 results
```

Stage 1 performs full-text BM25 scoring across venue names, tags, and reviews. Stage 2 boosts candidates that belong to the same k-means cluster as the user's query embedding, creating a personalized re-ranking effect.

### 7.2 Scoring Formula

The final re-ranking score combines the BM25 text relevance with cluster affinity:

```
finalScore = α × BM25Score + β × cosineSim(query, candidate) + γ × clusterBoost
```

Where:

| Parameter      | Value                              | Description                      |
| -------------- | ---------------------------------- | -------------------------------- |
| `α`            | 0.5                                | Text relevance weight            |
| `β`            | 0.35                               | Semantic similarity weight       |
| `γ`            | 0.15                               | Cluster affinity boost           |
| `clusterBoost` | 1.0 if same cluster, 0.0 otherwise | Binary cluster membership signal |

### 7.3 Dynamic K Selection

The optimal `k` depends on the number of saved venues. WorkSphere uses a heuristic:

```
k = min(max(floor(√n / 2), 3), 20)
```

where `n` is the number of user vectors. This ensures at least 3 clusters and at most 20, scaling with the square root of the data size.

## 8. Local Privacy Guarantees

### 8.1 Threat Model

| Threat                                 | Mitigation                                         |
| -------------------------------------- | -------------------------------------------------- |
| Server learns raw user vectors         | Only differentially-private centroids are synced   |
| Centroid inversion attacks             | Laplace noise added to shared centroids            |
| Cross-user inference                   | Per-user clustering; no cross-user data mixing     |
| Browser fingerprinting via cluster IDs | Cluster IDs are local-only; never leave the device |

### 8.2 Differential Privacy on Shared Centroids

When centroids are synced to the server for global aggregation, calibrated noise is added to satisfy ε-differential privacy.

```typescript
function addLaplaceNoise(
  centroid: Float32Array,
  sensitivity: number,
  epsilon: number,
): Float32Array {
  const noisy = new Float32Array(centroid.length);
  for (let i = 0; i < centroid.length; i++) {
    const scale = sensitivity / epsilon;
    // Inverse CDF of Laplace distribution
    const u = Math.random() - 0.5;
    const laplace = -scale * Math.sign(u) * Math.log(1 - 2 * Math.abs(u));
    noisy[i] = centroid[i] + laplace;
  }
  return noisy;
}

function dpCentroids(centroids: Float32Array[], epsilon = 0.5): Float32Array[] {
  // Sensitivity = max distance any single vector can shift a centroid
  // For normalized vectors in dim=1024: sensitivity ≈ 2/√1024 ≈ 0.0625
  const sensitivity = 0.0625;
  return centroids.map((c) => addLaplaceNoise(c, sensitivity, epsilon));
}
```

### 8.3 Privacy Budget Tracking

WorkSphere tracks the cumulative privacy budget per user using the composition theorem:

```typescript
interface PrivacyBudget {
  totalEpsilon: number;
  maxEpsilon: number; // default 2.0
  syncCount: number;
}

function canSync(budget: PrivacyBudget): boolean {
  return budget.totalEpsilon < budget.maxEpsilon;
}

function recordSync(budget: PrivacyBudget, epsilonUsed: number): PrivacyBudget {
  return {
    ...budget,
    totalEpsilon: budget.totalEpsilon + epsilonUsed,
    syncCount: budget.syncCount + 1,
  };
}
```

### 8.4 Privacy Benchmarks

| Metric                       | Value                | Notes                                             |
| ---------------------------- | -------------------- | ------------------------------------------------- |
| ε per sync                   | 0.5                  | Standard privacy guarantee                        |
| Max cumulative ε             | 2.0                  | Allows ~4 syncs before budget exhaustion          |
| Centroid L2 sensitivity      | 0.0625               | Bounded by normalized 1024-dim vectors            |
| Noise magnitude (Laplace)    | ~0.125 per dimension | At ε=0.5                                          |
| Inversion attack success     | < 2%                 | Tested with 10k synthetic vectors                 |
| Recommendation accuracy loss | < 3%                 | With ε=0.5 noise vs. clean centroids              |
| Sync frequency               | Every 6 hours        | Or on `visibilitychange` when tab is backgrounded |

## 9. IndexedDB Persistence

### 9.1 Schema

Centroids and cluster assignments are persisted in IndexedDB to survive page reloads and avoid re-clustering.

```typescript
// src/lib/clustering/idb.ts

const DB_NAME = "worksphere-clustering";
const DB_VERSION = 1;

interface ClusteringState {
  key: string; // "current"
  centroids: Float32Array[];
  assignments: Uint16Array;
  labels: string[];
  k: number;
  dim: number;
  updatedAt: number;
  privacyBudget: PrivacyBudget;
}

async function saveClusteringState(state: ClusteringState): Promise<void> {
  const db = await openDB(DB_NAME, DB_VERSION);
  await db.put("clustering", state, state.key);
}

async function loadClusteringState(): Promise<ClusteringState | null> {
  const db = await openDB(DB_NAME, DB_VERSION);
  return (await db.get("clustering", "current")) ?? null;
}
```

### 9.2 Sync Strategy

```
1. Page load → Load centroids + assignments from IndexedDB
2. User interaction → Update vectors → Re-cluster (incremental if k unchanged)
3. visibilitychange → If dirty, sync dp-centroids to server
4. Server response → Merge global centroids → Re-assign local vectors
5. Unload → Persist final state to IndexedDB
```

## 10. Inertia and Cluster Quality Metrics

### 10.1 Elbow Method

The worker reports inertia (total within-cluster sum of squares) for each `k`, enabling the elbow method for optimal `k` selection:

```
inertia(k) = Σ_{i=1}^{n} d(vectors[i], centroids[assignments[i]])²
```

The UI can plot `inertia(k)` for `k ∈ {2, 3, ..., 20}` and suggest the `k` at the elbow point where marginal inertia reduction drops below 10%.

### 10.2 Silhouette Score

The silhouette score measures cluster cohesion and separation:

```
s(i) = (b(i) - a(i)) / max(a(i), b(i))
```

where:

- `a(i)` = mean intra-cluster distance for vector `i`
- `b(i)` = mean nearest-cluster distance for vector `i`

A score near 1 indicates well-clustered data; near 0 indicates overlapping clusters; negative scores indicate potential misassignment.

## 13. Troubleshooting

| Symptom                          | Likely cause                            | Fix                                              |
| -------------------------------- | --------------------------------------- | ------------------------------------------------ |
| Worker timeout (> 30s)           | Vector count too large for single batch | Reduce batch size or increase `k`                |
| Empty clusters after convergence | `k` too large for data size             | Reduce `k` to `min(max(floor(√n/2), 3), 20)`     |
| Recommendation accuracy drop     | Privacy noise too high                  | Increase ε (e.g., 0.5 → 1.0) at cost of privacy  |
| Centroids not persisting         | IndexedDB quota exceeded                | Implement LRU eviction for old clustering states |
| Re-ranking returns empty         | No vectors indexed yet                  | Ensure `ADD_VECTORS` is called before `CLUSTER`  |
